# Create Transaction Idempotency Contract

## Status

Blueprint draft for owner review.

This document is not an implementation patch.

This document does not authorize migration, production code changes, UI changes, API changes, PostgreSQL work, Go work, or dashboard work.

## Purpose

Define the idempotency contract for create transaction workspace before implementing duplicate-submit prevention.

Current characterization proves that duplicate create workspace submits create duplicate transaction records.

The goal is to make create transaction safe against browser double-submit, retry, refresh-resubmit, network retry, and future API retry behavior.

## FACT

- Create transaction workspace route is the active create flow.
- Current duplicate submit behavior is characterized.
- Current behavior creates duplicate notes for duplicate create workspace submits.
- Existing create service-only lifecycle proof covers:
  - full cash payment
  - partial cash payment
  - no-payment debt/save-note
  - full transfer payment
  - partial transfer payment
  - note history projection
  - payment component allocation
  - auto-close on full payment
- Existing rollback proof covers forced failure after inline payment writes for service-only create flow.
- Full production-grade create transaction maturity is blocked by idempotency / duplicate submit prevention.

## GAP

- No idempotency key contract exists for create workspace.
- No idempotency persistence table is defined.
- No replay response policy is defined.
- No same-key different-payload rejection policy is defined.
- No TTL / retention policy is defined.
- No rollback behavior for failed idempotency attempts is defined.
- No concurrency proof exists for same idempotency key.
- No store-stock/inventory create lifecycle idempotency proof exists.
- No future API idempotency contract exists.

## ASSUMPTION

### A1 - Create workspace needs request-level idempotency

Assumption:

Create transaction workspace must accept an idempotency key generated by the client or server-rendered form.

Reason:

Duplicate submits can happen from browser double-click, refresh-resubmit, slow network retry, or future API retry.

Risk if wrong:

Without a key, the backend can only guess duplicate intent from payload similarity, which can reject legitimate same-customer transactions.

Containment:

Use explicit idempotency key, not fuzzy payload dedupe.

### A2 - Persistence should be DB-backed

Assumption:

Idempotency state should be persisted in a database table, not session-only memory.

Reason:

Transaction creation writes financial records and must survive process restarts, retries, and future API usage.

Risk if wrong:

Session-only idempotency may fail across devices, workers, APIs, or queue boundaries.

Containment:

Introduce a DB-backed idempotency table in a later migration slice.

### A3 - First implementation should target create workspace only

Assumption:

The first idempotency slice should cover `notes.workspace.store` only.

Reason:

Duplicate submit is proven in create workspace. Payment-after-note, revision, refund, and API paths need separate contracts.

Risk if wrong:

A broad idempotency framework could sprawl into unrelated flows and blur proof.

Containment:

Define generic enough table fields, but implement one route/use case first.

## DECISION

### D1 - Use explicit idempotency key

Create transaction idempotency must use an explicit key.

Rejected:

- fuzzy duplicate detection by customer name/date/amount
- hidden browser-only debounce as final protection
- session-only dedupe as final protection
- unique constraint on note customer/date/total

Reason:

Legitimate customers can have same date and same total. UI debounce is helpful but not financial truth. Backend must own the invariant.

### D2 - Idempotency scope

Idempotency key scope should include:

- actor id
- route / operation name
- idempotency key

Working operation name:

`create_transaction_workspace`

Reason:

The same key from the same actor should apply only to this operation, not to unrelated actions.

### D3 - Payload hash is required

Persist a canonical request payload hash.

Same actor + operation + key + same hash:

- return/replay previous successful result or redirect outcome.

Same actor + operation + key + different hash:

- reject with conflict.

Reason:

Same key with different payload is unsafe. Computers should not play “guess which transaction I meant”, because they are even worse at vibes than humans.

### D4 - Response replay policy

For Blade create workspace, replay should preserve user-facing behavior:

- if first submit succeeded, duplicate submit should redirect to cashier notes index with the same success-style outcome or a clear already-processed message.
- it must not create another note.
- it should expose the original note id internally for audit/debug.

Future API can return the original response body/status where appropriate.

### D5 - Failed attempt policy

If a create attempt fails before commit:

- idempotency record should not lock the key as successful.
- retry with same key and same payload should be allowed.

If a create attempt is processing:

- concurrent same-key request should not create a second transaction.
- acceptable first implementation can return a conflict/retry-later response, or block/serialize if transaction locking is implemented.

### D6 - Store successful result reference

Idempotency record should store at minimum:

- id
- actor_id
- operation
- idempotency_key
- request_hash
- status
- response_type
- result_note_id
- result_payload_json
- locked_at
- completed_at
- expires_at
- created_at
- updated_at

Allowed statuses:

- processing
- succeeded
- failed

### D7 - Do not patch production before tests

First implementation slice after this blueprint must start with RED tests.

Production patch is allowed only after:

- idempotency contract accepted
- migration shape selected
- target tests stated
- rollback/concurrency behavior scoped

## Proposed Table

Working table:

`idempotency_records`

Suggested columns:

- `id` string primary key
- `actor_id` string
- `operation` string
- `idempotency_key` string
- `request_hash` string
- `status` string
- `response_type` string nullable
- `result_note_id` string nullable
- `result_payload_json` json nullable
- `last_error` text nullable
- `locked_at` timestamp nullable
- `completed_at` timestamp nullable
- `expires_at` timestamp nullable
- `created_at` timestamp nullable
- `updated_at` timestamp nullable

Suggested unique key:

- unique actor_id + operation + idempotency_key

PostgreSQL readiness:

- use string status, not MySQL enum
- use string ids
- avoid JSON as the only query-critical field
- keep operation/key/status explicit

## Use Case Contract

Working service names:

- `CreateTransactionWorkspaceIdempotencyGuard`
- `IdempotencyRecordWriterPort`
- `IdempotencyRecordReaderPort`
- `DatabaseIdempotencyRecordAdapter`

Expected handler flow:

1. Resolve actor id and idempotency key.
2. Compute canonical request hash.
3. Begin idempotency claim for actor + operation + key.
4. If same key + same hash + succeeded:
   - return replay result.
5. If same key + different hash:
   - reject conflict.
6. If claim acquired:
   - run create transaction workspace inside existing transaction boundary.
7. On successful commit:
   - mark idempotency record succeeded with result note id / result payload.
8. On rollback/failure:
   - mark failed or clear claim according to selected retry policy.
9. Return normal result.

## Request Boundary

`idempotency_key`

Validation:

## Risks

- Idempotency record can become stale if failure policy is wrong.
- Replay response can confuse Blade flow if message/redirect contract is not explicit.
- Same-key different-payload behavior must reject safely, not silently return old result.
- If idempotency write is outside the right transaction boundary, rollback semantics can become misleading.
- If implementation uses only session, future API and worker retry behavior will diverge.
